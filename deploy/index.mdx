Nitro generates different output formats from the same codebase, making it suitable for any hosting provider. Built-in deployment presets adjust the output format with almost no additional code or configuration.

## How presets work

A deployment preset is a named configuration that tells Nitro how to build and package your application for a specific hosting environment. When you run `nitro build`, Nitro produces a self-contained `.output/` directory shaped for the target provider.

The default preset is `node_server`, which outputs a standard Node.js HTTP server. Other presets produce Cloudflare Workers bundles, serverless function packages, static site exports, and more.

## Setting a preset

**Via environment variable** (recommended for CI/CD):

```bash
NITRO_PRESET=cloudflare_pages nitro build
```

**Via CLI argument:**

```bash
nitro build --preset cloudflare_pages
```

**Via `nitro.config.ts`:**

```typescript nitro.config.ts
import { defineNitroConfig } from "nitro/config";

export default defineNitroConfig({
  preset: "cloudflare_pages",
});
```

## The `.output/` directory

After a successful build, Nitro writes all output to `.output/`:

```
.output/
├── server/         # Server entry point and bundled handler
│   └── index.mjs   # (or index.ts for Deno presets)
├── public/         # Static assets served directly by the CDN/server
└── nitro.json      # Build metadata
```

The entire `.output/` directory is self-contained. Deploy it to your hosting provider without needing `node_modules`.

## Zero-config providers

When deploying via CI/CD, Nitro auto-detects the provider environment and applies the right preset automatically. No extra configuration required for:

- [AWS Amplify](/deploy/providers/aws)
- [Azure Static Web Apps](/deploy/providers/azure)
- [Cloudflare Workers & Pages](/deploy/providers/cloudflare)
- [Firebase App Hosting](/deploy/providers/firebase)
- [Netlify](/deploy/providers/netlify)
- [StormKit](/deploy/providers/other)
- [Vercel](/deploy/providers/vercel)
- [Zeabur](/deploy/providers/other)

<Warning>
For Turborepo users, zero-config detection may be blocked by Strict Environment Mode. Explicitly allowlist the required environment variables or use Loose Environment Mode (`--env-mode=loose`).
</Warning>

## Compatibility dates

Deployment providers regularly update their runtime behavior. Nitro uses compatibility dates to let you lock in behavior at project creation time and opt in to future updates when you're ready.

When you create a new project, set `compatibilityDate` to the current date:

```typescript nitro.config.ts
import { defineNitroConfig } from "nitro/config";

export default defineNitroConfig({
  compatibilityDate: "2024-09-19",
  preset: "cloudflare_module",
});
```

Update the compatibility date periodically and test your deployment thoroughly after each update.
